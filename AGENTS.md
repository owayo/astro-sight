# astro-sight

AI エージェント向け AST 情報生成 CLI (Rust)

## Architecture

- **AppService 層** — CLI / Session / MCP の統一コアロジック（`src/service.rs`）
- **tree-sitter** ベースの構文解析エンジン（16言語対応）
- **手書き lexer** バックエンド（`src/engine/lexer.rs`）。tree-sitter で OOM する言語向けの fallback として v26.6 で導入。現状 Xojo を lexer-only でサポート（`symbols` / `refs` / `dead-code` のみ動作、`calls` / `imports` / `ast` / `lint` / `sequence` は `UNSUPPORTED_LANGUAGE` を返す）。modifier ループは複数修飾子を順番に消費する（`Public Shared Sub Foo()` のような可視性 + Shared/Static 併記の標準記法を `Shared` 消費後に定義 keyword と突き合わせる、v26.6.110）。同一 keyword の重複は consumed リストで除外して病的入力でも無限ループしない
- **BLAKE3** ベースのファイルキャッシュ（単一ファイル `ast` / `symbols` は内容ハッシュに canonical path と astro-sight バージョンを混ぜ、同一内容の別ファイルや別言語で `path` / `lang` が混線せず、バージョン更新時は解析ロジック/出力スキーマの変更に追従して旧キャッシュが自動失効する＝内容不変でも結果が変わるケースで stale を防ぐ。書き込みは同一ディレクトリ temp + rename の atomic 方式で、kill / 電源断でも truncated JSON が残らない。読み出しは末尾 `}` の軽量検証付きで、不正エントリは miss 扱い + 自己削除。`cmd_ast` / `cmd_symbols` は put 前に service が実際に解析した内容の hash (`response.hash`) と cache key の元になった read 内容の hash を照合し、2 回の read の間にファイルが更新された場合は put しない＝hash(旧内容)→output(新内容) の TOCTOU キャッシュ汚染を防ぐ）
- **clap derive** による CLI 引数パーサー
- **NDJSON** ストリーミングセッション対応
- **スマートコンテキスト**（diff→影響分析、関数シグネチャ変更は識別子境界で照合して prefix 名の別関数を誤検出しない、単一/バッチ参照検索は fold/reduce でピーク RSS を抑制、バッチ参照検索 O(N+S)、`find_references_batch` はディレクトリ走査と rayon pool を 1 回に集約し内部で名前を chunk 分割=既定 64・`ASTRO_SIGHT_REFS_BATCH_CHUNK` で調整して AC trie / fold バケットのメモリを上限化＝呼び出し側の chunk 毎再走査をなくし、chunk サイズに依らず結果は一致する。Angular テンプレート参照は chunk ループ前に `AngularBatchContext::prepare(dir, glob)` で `is_angular_project` 判定 + `collect_component_templates` を 1 度だけ実行して chunk 横断で再利用する（v26.6.110、chunk 数倍の全 dir/.ts 走査を 1 回に削減。非 Angular リポでは `None` で完全 skip））
- **AST 応答の巨大行抑制**（`ast` の `text` / `snippet` は 256 文字上限で切り詰め、巨大行で応答が肥大化しない）
- **`.h` ヘッダの C/C++ 自動判定** — `.h` は既定では C として扱うが、C++ 専用構文マーカーがあり、C++ parser の方が C parser より parse error が少ない場合だけ C++ に切り替える。通常の C ヘッダを不用意に C++ 扱いせず、`class Foo { public: ... }` や `struct X : Base<X> {}` のような C++ ヘッダの `symbols` / `review` / `dead-code` 取りこぼしを抑える
- **MCP サーバーモード**（rmcp 2.2 による stdio JSON-RPC 2.0、ワークスペースサンドボックス、11 ツール。rmcp 2.x では `Content` が `ContentBlock` enum に統合された）
- **デフォルト compact JSON** 出力（`--pretty` で整形出力）
- **バッチ処理**（`--paths` / `--paths-file` で複数ファイル NDJSON 出力）
- **JSON エラー出力**（`{"error":{"code":"...","message":"..."}}` を stdout に出力。stdout pipe が先に閉じた場合は broken pipe panic を出さず exit 0 で終了）
- **入力検証の強化**（`refs` の空 `name/names` を拒否、`--paths` / `--paths-file` の空リストを拒否、`--paths-file` は 100MB 上限付きで読み込み、`cochange` の `min_confidence` / smoothing prior は有限範囲に制限、`session` の空文字・非 UTF-8・不正パスの `ASTRO_SIGHT_WORKSPACE` を拒否、`--dir` にはディレクトリのみ許可、streaming `context` は JSON prefix 出力前に入力を検証）
- **セキュリティ** — パス境界チェック（Session/MCP のワークスペースサンドボックス、相対パスはワークスペースルート基準、非 UTF-8 パスは canonicalize 後に fail-closed で拒否、`review --diff` 経由の `detect_api_changes` / `filter_diff_files_for_dead_code` も diff の `new_path` / `old_path` を `is_safe_diff_path` でトラバーサル検証。`should_skip_diff_file` は `is_safe_diff_path` の文字列 check に加えて、新ファイルを `canonicalize().starts_with(canonical_dir)` で fail-closed 判定し、`evil.rs -> /etc/passwd` のような symlink で workspace 外を指す追加ファイルからの識別子リークを阻止する）、ファイル/入力サイズ 100MB 上限（`session` / `context` / `impact` の生入力、`--paths-file` を含む、`parser::read_file` は metadata 取得後に拡大する TOCTOU 経路も `take()` + 読み込み後再検証で阻止）、`git diff` / `git show` / `git blame` に渡す `--base` 等 revision は `-` プレフィクス / NUL / 空文字を拒否（オプション誤認識防止、`cochange --blame` の `--paths` / `--paths-file` 経由でも検証）
- **非 ASCII ファイル名対応** — ファイル名を stdout から解析する git 呼び出し（`run_git_diff` / blame source の `diff --name-only` / cochange の `diff-tree --name-only` / `changed_new_lines_for_file`）は `-c core.quotepath=off` を付け、日本語等のファイル名が 8 進クォート (`"\346..."`) されて diff ヘッダ照合が崩れるのを防ぐ。防御として `parse_unified_diff` は認識できない `--- ` / `+++ ` ヘッダ行で state を flush/リセットし、quotepath 形式の外部 diff でも hunk が直前ファイルへ誤帰属しない（該当ファイルは解析対象外に落ちる fail-safe）
- **トークン最適化** — version フィールド省略（doctor/MCP のみ）、compact キー短縮（`lang`/`ln`/`col`/`ctx`/`refs`/`src`/`def`/`ref`/`fn`/`cx` 等）、calls を caller グルーピング、CompactAstEdge フラット化、refs/context で相対パス出力、symbols デフォルト compact 出力（`--doc` で docstring 付加、`--full` で旧来の完全出力、`-q/--query` でカスタム tree-sitter クエリが built-in を置換 — capture 名は `function.name` / `class.name` 等の既存語彙に限定し、不正クエリ・未知 capture・lexer-only 言語は `INVALID_REQUEST`。cache key には query hash を混ぜ、default query の key は不変。`--path` / `--paths` / `--dir` / session の `query` フィールドで有効）
- **dead-code 規約除外 / member liveness** — PHPUnit 規約 (`*Test` / `*TestCase` クラス、`testXxx` / `setUp` / `tearDown` 等) と Python unittest/pytest 規約 (`unittest.TestCase` 派生クラス + 同一ファイル内の間接継承を fixed-point で解決、`test_*` / `setUp` / `tearDown` / `setUpClass` / `tearDownClass` / `addCleanup` / `addClassCleanup` メソッド、`test_*.py` / `*_test.py` のトップレベル `test_*` 関数、`conftest.py` 内の関数) を dead-code から除外（テストランナーが動的 discover するため）。Angular `@Component` / `@Directive` 装飾クラスのライフサイクルフック (`ngOnInit` / `ngOnDestroy` / `ngOnChanges` / `ngDoCheck` / `ngAfterContentInit` / `ngAfterContentChecked` / `ngAfterViewInit` / `ngAfterViewChecked`) も Angular ランタイムが change detection サイクルで自動呼び出しする暗黙エントリポイントのため dead-code から除外。Xojo (lexer-only) では `#tag Event` 〜 `#tag EndEvent` 配下のイベントハンドラ (`Sub` / `Function`、Xojo ランタイムがイベント駆動で呼ぶ) と、`Inherits TestGroup` を直接/同一ファイル内間接 (fixed-point) に継承するクラスの引数なし `*Test` / `Setup` / `TearDown` メソッド (XojoUnit が Introspection で列挙・実行) を runtime entrypoint として dead-code から除外 (`lexer/xojo.rs::runtime_entrypoint_lines`、定義行 (range) で照合し bare name 除外による汎用名 (`Action` / `KeyDown`) の巻き込みを回避、dead-code 経路のみで API 差分では公開 API 面として残す)。TS/JS と PHP の duplicate な class member は owner を安全に一意推定できる場合だけ従来の bare name 重複スキップを緩和する。TS/JS は **receiver-aware 帰属**: `Owner.member` (static access — **import または同ファイル class 定義に限定**、単なる owner 名一致では確定しない) / `new Owner().member` / owner 型 annotation・`new Owner()` 初期化子付き変数 / `Owner[]` 配列変数の `xs[i].member` / enclosing class が owner の `this.member` だけを確定票とし、factory・DI・関数戻り値など receiver を静的に辿れない access が 1 件でもあれば set 全体を Ambiguous に倒す (import 有無ベースの帰属は廃止)。同名のローカル binding があれば shadow とみなし変数として判定する (`const Alpha = getBeta(); Alpha.fmt()` を class access と誤認しない)。`import * as ns` 経由のアクセス (`ns.Alpha.fmt()`) は receiver が member_expression になり通常分類で不解決に落ちるため、namespace import の存在だけでファイル全体を Ambiguous にはしない。PHP は `Owner::method()` と同一クラス内 `self::method()` を確定参照として数え、`$obj->method()` や callable 文字列など owner 不明の参照があれば ambiguous として旧スキップへ戻す。`__construct` の duplicate set は `new Foo()` (object_creation_expression) をクラス名票として数える (`new $var()` / `new static()` / `new parent()` は Ambiguous、bare 名 prefilter は `new` を含むファイルも parse 対象に残す)。`use X\Y as Z;` の alias (grouped use 含む) はファイル単位で収集して `Z::m()` / `new Z()` を実クラスへ解決する (単一 namespace + 競合なしで一意解決できた場合のみ。multi-namespace や競合 alias では alias 名への参照だけ Ambiguous に倒し silent Ignore の dead 誤検出を防ぐ)。`static::` は遅延静的束縛でサブクラス override へディスパッチされ得るため `parent::` と同様 Ambiguous に倒す。trait `use` の合成先 (class / trait / enum、PHP 8.1+) を graph 化して `UseSite::method()` を一意到達の trait method へ解決するが、合成先が同名の**具象**メソッドを自己宣言している場合は PHP の解決順（自クラス > trait）により trait 側へ辿らない（abstract 宣言は trait 実装で満たされるため通常どおり辿る）。member liveness / trait use 収集の走査集合には hidden ディレクトリ配下の diff 候補ファイルも `refs::merge_extra_files` で合流させ、count 経路と一致させる。Angular member decorator（`@HostListener` 等）とメソッドの対応付けは間に挟まったコメントを読み飛ばす。`--dead-scope touched-symbols` は qualname (`Container.method`) の宣言行を bare name fallback で解決し、class member にもフィルタが効く
- **C/C++ dead-code liveness 補正** — 本体を持たない前方宣言 / opaque tag (`typedef struct st_mysql MYSQL;` の `st_mysql` 等、外部ライブラリの不透明構造体タグ) は「定義」ではないため dead 候補から除外 (`symbols.rs::is_cpp_forward_declaration`)。body あり `struct` / `class` / `union` / `enum` tag は定義行だけ Definition とし、`struct X *p` / `struct X value` / `sizeof(struct X)` / cast / 関数引数やメンバ宣言内の tag 名は `refs` / dead-code の count-only 経路で Reference として数える。enum は型名が直接使われなくても列挙子のいずれかが参照されていれば live、body あり typedef tag は alias 名経由の参照でも live と判定する (`symbols.rs::collect_cpp_dead_liveness_aliases` で enum→列挙子 / typedef tag→alias を集め、`refs.rs::cpp_typedef_enum_definition_context` で enumerator name / typedef alias 名のみ Definition 分類し宣言行を参照と二重計上しない。typedef の元型・enumerator value 式・配列長式内の識別子は Reference)。すべて `LangId::C | Cpp` で閉じ symbols 公開出力は不変
- **C/C++ メソッド・ポインタ返り関数の抽出** — C++ クラスメソッド (`field_identifier`)、クラス外定義 (`qualified_identifier` の末尾名)、C/C++ のポインタ返り (`pointer_declarator`、`Type *foo()`) / C++ 参照返り (`reference_declarator`、`T& m()`) 関数を `symbols` / `calls` / `dead-code` / `impact` で抽出する (旧クエリは `function_definition > function_declarator` 直結のみで、宣言子が包まれるポインタ返りやメソッドを取りこぼしていた)。クエリは `function_declarator` 配下の名前を直接キャプチャし `cpp_enclosing_function_definition` で本体まで辿るため、本体の無い宣言 (プロトタイプ) は採用しない。C++ メソッドは可視性 (`access_specifier` の public/protected/private、class デフォルト private・struct デフォルト public) を `symbols.rs::is_exported_cpp` で判定し、非公開メソッドの変更を API 差分の偽陽性にしない (自由関数・クラス外定義は保守的に公開扱い。access_specifier は field_declaration_list の直接子なので「parent が field_declaration_list である祖先」= member_anchor から prev_sibling を遡る — template メンバは `field_declaration_list > template_declaration > function_definition` 構造のため method 自身からの遡りでは届かず private 誤判定になる)。C/C++ 関数の doc comment は昇格した function_definition 自身の直前コメントから拾う (name の親 = function_declarator の prev sibling は戻り型で comment に届かない)。Ruby の `class A::B` / `module A::B` は name capture が scope_resolution 内で range が名前に潰れるため class / module ノードへ昇格し、本体全体を range にして配下メソッドの container 帰属を成立させる。`calls` の caller 名は宣言子を末尾識別子まで辿り `foo(args)` のような汚染を防ぐ (`calls.rs::c_function_name_from_declarator`)
- **フレームワーク自動検出** — `--framework` 未指定時でも `<dir>/package.json` の `dependencies` / `devDependencies` に `next` キーがあれば自動で `nextjs` プリセットを適用 (`peerDependencies` / `optionalDependencies` は誤爆しやすいため対象外)。明示指定は常に auto detect に優先する
- **API 差分検出 (api.add / api.rm / api.mod)** — bin-only Rust crate (`src/lib.rs` なし & `Cargo.toml` の `[lib]` セクションなし) の `pub fn` 追加・削除・シグネチャ変更は外部公開 API ではないため API 差分から除外。`api.rm` 経路は base リビジョン側の `Cargo.toml` / `src/lib.rs` を `git show` で取得して判定し、新ツリーで `src/lib.rs` を削除した同時 diff でも旧公開 API の削除は誤抑制しない。C/C++ では tree-sitter が誤パースする実関数 body 内ネスト `function_definition` (`BOOST_FOREACH(...){}` 等のマクロ呼び出し) を exported から除外し (`symbols.rs::is_cpp_nested_function`)、同名シンボルが旧/新いずれかに複数ある場合 (overload / 誤パース) は別物同士の突き合わせを避けるため曖昧として api.mod から除外する。TS/JS では named export clause で公開が継続しているシンボル — `export { foo } from "./b"` (forwarding) に加え、`import type { X } from "...";` + `export type { X };` の **from 句なし re-export** も — は利用者から見た export 面 (import path から取れる名前) が維持されるため `api.rm` から除外する (`symbols.rs::collect_js_ts_named_export_surface_names` が from 句の有無を問わず export clause の公開名を抽出 (alias 対応)。ローカル定義付き `export { X }` の X も集合に入るが、定義が exported symbols として別途現れ api.rm 候補に上がらないため無害。`has_named_export` は from 句付き export をローカル export 判定から除外し、両者は `visit_js_ts_export_clause_specifiers` visitor で AST walk を共用)。ローカル定義を別モジュールへ切り出し re-export を残すリファクタの誤検出を防ぎ、`reconcile_with_moves` (move 相殺、name+kind+signature 完全一致) とは独立に効く — 定義移動時に定義形も変わる (union literal → `keyof MessageMap` 等) と move 相殺は signature 不一致で不成立のため、この抑制が唯一の防御になる。`export * from` (wildcard) は静的に名前を解決できないため抑制対象外 (fail-open 回避)。`filter_exported_symbols` の `pub(crate)` 除外テキスト判定は Rust 限定（他言語の `pub` という名前の関数が API 面から消えるのを防ぐ）。Rust の**モジュール再エクスポート** (`pub use internal::wifi as api;` 等) は `compute_reexport_reachable_modules` が `pub mod` 経路の public_modules を seed に「Named edge の source が到達可能なら target module とその pub 子孫も到達可能」の固定点で `reachable_modules` を構築し、`exposes_symbol` はこの集合で公開判定する（旧実装は resolved_prefix が空の use edge を silent drop しており、module 再エクスポート経由の pub fn 削除・変更が完全無音になっていた）。到達可能化の条件は target が (a) `collect_all_modules` = private にも潜る全 module walk に実在し、(b) **親 module で `pub mod` と宣言されている**こと — private module 自体の再エクスポート (`mod wifi; pub use self::wifi as api;`) は rustc の E0365 で無効なため API 面にしない。module walk の子ファイル解決は `child_module_base_dir` で lib.rs/main.rs/mod.rs (自身の dir) と file-style module (`internal.rs` → `internal/` 配下) を区別する (旧実装は parent dir 固定で `internal.rs` 内の `pub mod api;` を `api.rs` と誤解決し階層 module を取りこぼしていた。既存 `collect_public_pub_mods` の同バグも修正)。`has_cross_file_refs` は bare 名で照合（qualname `Container.method` は identifier に一致せず常に 0 件となり「自ファイル内で呼ばれているだけ」で api.mod 除外が成立してしまうため）。`detect_python_property_to_field` は削除元ファイルが Python の場合のみ適用。非破壊的な互換変更は `compatible_modified` に分離し、React HOC ラップ、未参照 object member 削除（destructuring `const { x } = config` も member 利用として数え、実利用のある削除は blocking 維持）、TS/TSX トップレベル関数の末尾 (`trailing_optional_params`、`commands/api_changes/python_signature.rs`、デコレータ差分や同名関数複数定義は false negative 回避のため保守的に blocking 維持) と、それらと同じシンボルに紐づく hook `impacts` は Stop hook の blocking 対象から外す。cross-file 参照判定 (`has_cross_file_refs` / `is_used_in_diff_paths` / `is_removed_symbol_unreferenced` / `is_modified_closed_in_diff` / `has_blocking_value_usage`) は `detect_api_changes` 前段 (Phase 0) で候補 name を収集し **ApiRefIndex** (`find_references_batch`、chunk 既定 64) に集約して per-symbol の全リポジトリ走査を 1 回のディレクトリ走査に削減 (候補は AC trie メモリ上限のため chunk 分割するが、`find_references_batch` が走査と pool を 1 回に集約するため chunk `ApiClosureCaches` を全 modified シンボル横断で共有し、per-symbol の `git diff` 起動と tree-sitter parse を unique file 単位 (O(F)) に削減する (v26.6.110、`classify_signature_change` → `is_modified_closed_in_diff` を `&mut ApiClosureCaches` で連鎖)
- **循環的複雑度** — symbols 出力に `cx`（cyclomatic complexity）を付加（関数/メソッドのみ、ベース1 + 分岐ノード数、ネスト関数/クロージャは除外）、言語別分岐ノード定義（Rust/JS/TS/Python/Go/Java/Kotlin/Ruby/PHP/C#等）。分岐ノードは **named ノードのみ**計上する（tree-sitter-ruby は `if` 文ノードとキーワードトークン `if` が同名 kind のため、named 制約が無いと `if` / `case` / `when` / `while` 等が二重計上され `cx` が約 2 倍になる。他言語の分岐ノードは全て named なので無影響）。Kotlin は tree-sitter-kotlin 固有のノード名（`if_expression` / `when_expression` / `when_entry` / `do_while_statement` / `catch_block` / `elvis_expression`）を使うため、Java と同じスライスを共用すると一切マッチせず `cx` がベース 1 のまま返る（v26.6.110 で別スライス化、`function_boundary_kinds` にも `anonymous_function` を追加）。Swift も同型で専用スライス（`if_statement` / `guard_statement` / `switch_statement` / `switch_entry` / `for_statement` / `while_statement` / `repeat_while_statement` / `catch_block` / `ternary_expression`）を使う — 汎用スライスには `guard_statement` / `switch_entry` が無く guard と case arm が計上されない。C/C++ は関数名を `function_declarator` 配下でキャプチャするため、シンボル抽出時に本体を持つ定義ノード (`function_definition`) まで繰り上げて (`symbols.rs::cpp_enclosing_function_definition`) range と複雑度を本体全体から算出する。`pointer_declarator`（`Type *foo()`）/ `reference_declarator`（`T& m()`）/ `qualified_identifier`（クラス外定義 `Foo::bar`）を経由しても辿り、本体の無い宣言（プロトタイプ・関数ポインタ）は採用しない（繰り上げないと range が宣言子行に潰れ複雑度が常に 1 になり、impact が関数本体のみの変更を取りこぼす）
- **設定ファイル** — `~/.config/astro-sight/config.toml`（TOML 形式、`astro-sight init` で生成、`log_path` 未指定時は config ファイル隣の `logs/`、明示指定時は同じ値でも尊重）
- **ロギング** — logroller による日次ローテーション（ローカルタイムゾーン、3日保持）。debug 時のログ初期化失敗（書込不可ディレクトリ等）は警告に留めて解析本体は継続する（read-only 環境などへの堅牢化）
- **git 管理外ディレクトリの graceful skip** — `--git` を受けるコマンド（context / impact / review / dead-code / cochange）が git 管理外 dir で実行された場合、`git rev-parse --is-inside-work-tree`（`LC_ALL=C`）で事前判定し「解析対象なし」として **exit 0** で skip。`--hook`（review / impact）は完全無出力（silent skip）、通常 CLI は各結果型に `skipped: Option<SkipInfo>`（`reason="not_git_repository"` / `source="git"` / `message`）を付けた空結果を返し「差分なし」と「git 管理外」を区別可能にする。真のエラー（base 不正・git 実行不能・壊れた repo・権限）は従来どおり exit 1（fail-closed）。判定は `commands.rs` の `resolve_git_diff`（経路A: context/impact/review/dead-code）と `resolve_blame_source_files`→`BlameSourceResolution`（経路B: cochange、`--paths`/`--paths-file` 明示時は管理外でも明示分を尊重）に集約。impact は構造化 JSON 出力を持たないため skip も無出力 exit 0（既存の「差分なし」と一貫）
- **未追跡 rename ペアリング** — unstaged の「ファイル削除 + 同内容の untracked」を high-confidence rename として合成 diff (`Modified{old,new}`) に正規化する（`git_input.rs::pair_untracked_renames`）。構造 gate は同一言語・サイズ比 2 倍以内・exported symbol 集合 (name,kind,signature) の完全一致で、両端 degree 1 の一意対応のみ採用（ambiguous は fail-closed で pair しない）。exported symbol の抽出は per-file の `RenameFacts` として **D+U 回だけ**事前計算し、相手側集合に同じ言語が無いファイルは parse 自体を省く（旧実装は D×U の全組合せで gate 内フルパースし、ディレクトリ移動 D=U=200 で数万回の再パースが起きていた。出力は旧実装と完全一致）

## Review Standards

- 修正対象は再現可能で根拠を示せる不具合に限る。推測や好みに基づく変更は行わない
- セキュリティ境界は fail-open にしない。設定値やサンドボックスが不正なら明示的に失敗させる
- コード変更の前には `astro-sight context --dir . --git`、変更後には `astro-sight impact --dir . --git` を実行する
- diff / PR 全体のレビュー依頼では、個別コマンドを積み上げる前に `astro-sight review --dir . --git` で全体像を確認する
- diff 全体の一括レビューでは `astro-sight review --dir . --git` も併用し、影響・共変更・API 差分・死蔵シンボルをまとめて確認する
- 公開 API や export を触った変更では `astro-sight dead-code --dir . --git` も併用して死蔵シンボルを確認する
- 構造確認が 2 手以上続くと分かっている場合（`symbols` → `imports` / `calls` / `sequence` 等）は最初から `session` にまとめ、プロセス起動コストと手順漏れを抑える
- `symbols` の 1 行情報だけで判断できない構文境界・parse エラー・巨大行の扱いは、Read/Grep ではなく `astro-sight ast --path <file> --line <n> --col <n>` で exact node を確認する
- 関連ファイルの変更漏れ確認は `review --dir . --git` の `missing_cochanges` または `cochange --dir . --paths <file>` を先に使い、ファイル名の勘だけで判断しない
- 繰り返しの構造ルール確認には `astro-sight lint --path <file> --rules <rules.yaml>` を使い、アドホックなテキスト検索で代用しない
- 並列集約や大規模リポジトリ向けの変更では `/usr/bin/time -l` でピーク RSS を測定し、`par_iter().collect()` で不要な中間 Vec を全展開していないか確認する
- コードコメントは必要な箇所にだけ付け、付ける場合は日本語で簡潔に記述する

## Lexer-Only バックエンド (v26.6 で導入)

### 背景
tree-sitter で実用的に parse できない言語向けに、手書き lexer による fallback バックエンドを持つ。
case-insensitive な GLR バックトラッキング系の tree-sitter grammar は parse 中に 1GB/秒級でメモリが線形に膨張する事案 (Xojo 4 行 diff で 30GB OOMKilled の経緯あり) のため、tree-sitter-xojo は v26.6 で Cargo 依存ごと削除した。

### 言語モデルの分離
`src/language.rs` で型を 3 つに分けている:
- `TreeSitterLang` — `ts_language()` を持つのはこの型だけ。Rust/Python/JS 等 16 言語
- `LexerLang` — 手書き lexer で解析する言語 (現状 `Xojo`)
- `DetectedLang` — `TreeSitter(TreeSitterLang) | LexerOnly(LexerLang)` の判定結果型

`LangId::ts_language()` を呼ぶ前に `is_lexer_only()` で振り分ける義務がある (lexer-only に対する `ts_language()` は panic する)。

### コマンド別の挙動
lexer-only 言語に対する各コマンド:
- `symbols` / `refs` / `dead-code` — `src/engine/lexer.rs` 経由で動作 (定義ヘッダ列挙 + identifier 走査)。`dead-code` は Xojo の `#tag Event` ハンドラと `Inherits TestGroup` 派生クラスの引数なし `*Test` / `Setup` / `TearDown` を runtime entrypoint として除外する (`lexer/xojo.rs::runtime_entrypoint_lines`)
- `ast` / `calls` / `imports` / `lint` / `sequence` — `UNSUPPORTED_LANGUAGE` エラーを返す
- `context` / `impact` / `review` — 全 changed file が lexer-only のみの diff では cross-file 解析を skip する。`review` は `dead_symbols` も含めて空結果を返す (lexer 経路の cross-file refs は汎用名 noise が多いため将来 PR で本格対応)

### 強制無効化（デバッグ用）
context / impact / review / dead-code の skip 機構は次の env で解除できる (本番では設定しない):
- `ASTRO_SIGHT_FORCE_CI_LANG_IMPACT=1` — context / impact / review 全体 skip / impact pass2 で skip 解除
- `ASTRO_SIGHT_FORCE_CI_LANG_DEAD_CODE=1` — dead_code で skip 解除 (v26.6 では dead-code 自体は lexer 経路で動くため deprecate 扱い)

### 言語追加時の手順
tree-sitter で動かない言語を追加するには:
1. `language.rs` の `LexerLang` に variant を追加
2. `LangId` も対応する variant を追加し `detected()` を更新
3. `src/engine/lexer.rs` に `LexerProfile` を新規追加 (keywords / コメント / 文字列区切り / 定義 keyword)
4. `profile_for()` の match arm に追加
